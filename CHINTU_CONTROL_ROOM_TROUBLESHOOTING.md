# Chintu Control Room — Troubleshooting

**Stage:** 11 (cycle 4)
**Audience:** founder + any future Claude/Codex session

Symptoms-to-fixes for the local control room: the static HTML index,
the agent dashboard, the alive briefing, and the master launcher. Each
fix here is read-only or rerun-only. None of these instructions push,
edit BALA app files, or call the network.

---

## A. "I opened `CHINTU_CONTROL_ROOM_INDEX.html` and the snapshot looks old"

The HTML index is a static snapshot regenerated on demand.

```powershell
powershell -ExecutionPolicy Bypass -File scripts\chintu-control-room-index.ps1
```

Then reload the file in your browser (Ctrl+R / Cmd+R). If the dashboard
also looks stale:

```powershell
powershell -ExecutionPolicy Bypass -File scripts\chintu-agent-dashboard.ps1
```

The cleanest single command is the master launcher, which regenerates
both at the end:

```powershell
powershell -ExecutionPolicy Bypass -File scripts\chintu-master-launcher.ps1
```

---

## B. "The master launcher halts on step N"

Each step is a real script. The launcher prints the failing step's
name. Reproduce the failing step in isolation:

| Step | Reproduce | Common cause |
|---|---|---|
| `node --check app.js` | `node --check app.js` | Founder edited BALA app code with a typo. Stop and ask. |
| snapshot consistency | `node scripts\chintu-snapshot-consistency.test.js` | Committed history out of sync with regenerated dashboard. Re-run the dashboard script. |
| command map integrity | `node scripts\chintu-command-map.test.js` | New script added without being listed in `CHINTU_FOUNDER_COMMAND_MAP.md`, or vice-versa. |
| memory vault integrity | `node scripts\chintu-memory-vault.test.js` | Vault file added without an entry in `CHINTU_MEMORY_VAULT/README.md`. |
| doc link integrity | `node scripts\chintu-doc-link-integrity.test.js` | A `CHINTU_*.md` / `BALA_*.md` link points to a missing relative path. |
| no network egress | `node scripts\chintu-no-network-egress.test.js` | A Chintu script gained a `fetch(`, `POST`, webhook, etc. Revert. |
| medical claims | `node scripts\chintu-medical-claims.test.js` | A doc gained "diagnose", "cure", "predict", etc. about BALA. Revert. |
| safety boundary | `node scripts\chintu-safety-boundary.test.js` | The protected BALA file list drifted from the canonical list. |
| validate.ps1 | `scripts\chintu-validate.ps1` | Manifest, privacy, or medical claims gate. |
| release-guard.ps1 | `scripts\chintu-release-guard.ps1` | Aggregates the above. |

## C. "The alive briefing or OS health check looks empty"

These two files are regenerated by:

```powershell
powershell -ExecutionPolicy Bypass -File scripts\chintu-alive-briefing.ps1
powershell -ExecutionPolicy Bypass -File scripts\chintu-os-health-check.ps1
```

If still empty, the script likely failed silently — re-run with
`-Verbose` if the script supports it, or check for a missing input
file (most often `CHINTU_MEMORY_VAULT/DAILY_LOGS/<today>.md`).

---

## D. "I clicked a link in the control room and got a 404"

The control room HTML links to local files relative to the repo root.
If a link is dead:

1. Confirm the target file exists: `Get-ChildItem <target>`.
2. If the file was renamed, regenerate the control room. The generator
   reads the founder command map and produces fresh links.
3. If the file was deleted on purpose, also remove the corresponding
   row from `CHINTU_FOUNDER_COMMAND_MAP.md` so the command-map test
   doesn't fail on the next launcher run.

---

## E. "I see `CHINTU_AGENT_DASHBOARD.html` and `CHINTU_OS_HEALTH_CHECK.md` in `git status` even though I didn't touch them"

Expected. They are generated-but-tracked artifacts (see
`CHINTU_GENERATED_FILES_MAP.md` section 2). The only diff is usually a
timestamp + the latest commit hash. Two safe responses:

- Commit them with `chore: refresh control room snapshots` if you're
  about to commit anything else.
- Leave them; the next launcher run will regenerate them and the diff
  will keep on rolling forward harmlessly.

---

## F. "I want to add a new script and have it show up in the control room"

Three edits in one commit:

1. Add `scripts/chintu-<name>.ps1` (or `.test.js`).
2. Add a row to `CHINTU_FOUNDER_COMMAND_MAP.md` in the right section.
3. If it produces a generated file, also add a row to
   `CHINTU_GENERATED_FILES_MAP.md`.

Then run the master launcher. The command-map test, the
generated-files-map test (when present), and the doc-link test will
catch any mismatch.

---

## G. When in doubt

Three commands triage almost any control-room problem:

```powershell
git status --short
powershell -ExecutionPolicy Bypass -File scripts\chintu-master-launcher.ps1 -SkipReports
powershell -ExecutionPolicy Bypass -File scripts\chintu-next-action.ps1
```

If the launcher is green and `next-action` has a suggestion, follow it.
If the launcher is red, fix the failing step before any other work.
