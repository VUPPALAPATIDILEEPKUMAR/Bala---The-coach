# Chintu Repo Hygiene Report

**Stage:** 11 (cycle 4)
**Mode:** observational; no auto-fixes

A founder-facing audit of the working-tree shape: what is well-kept,
what is drifting, and what would be worth cleaning up safely on a slow
day. This is a docs-only report. Nothing here triggers any change.

---

## 1. What "clean" means here

The repo is clean when:

- `git status --short` shows only regenerated artifacts (tracked HTML
  dashboards + alive briefing + OS health check). See
  `CHINTU_GENERATED_FILES_MAP.md` section 2.
- All `CHINTU_*.md` and `BALA_*.md` markdown links resolve
  (`chintu-doc-link-integrity.test.js`).
- The founder command map matches `scripts/` 1:1
  (`chintu-command-map.test.js`).
- The memory vault README matches the vault filesystem
  (`chintu-memory-vault.test.js`).
- No script touches the network or claims to diagnose / cure
  (`chintu-no-network-egress.test.js`, `chintu-medical-claims.test.js`).
- The protected BALA file list in `CHINTU_CLAUDE_SURVIVAL_HANDOFF.md`,
  the survival rules in `chintu-safety-boundary.test.js`, and the
  command map are all in agreement.

Each of those is enforced by a test that runs from the master
launcher. If any of them is red, that is the first thing to fix.

---

## 2. Known healthy patterns (keep these)

- **One safe slice per commit.** Cycle 2 and cycle 3 stuck to this
  faithfully; the commit log reads as a clean progression.
- **One docs file per concept.** `CHINTU_START_HERE.md`,
  `CHINTU_WHEN_STUCK.md`, `CHINTU_ARTIFACT_POLICY.md`, etc. are short,
  named for their purpose, and link to each other. Avoid merging them.
- **Tests live next to scripts.** `scripts/chintu-*.test.js` for every
  major operator surface. Wire each into the release guard.
- **Generators are idempotent.** Running the master launcher twice in a
  row produces a stable working tree (only timestamp/commit-hash
  diffs).

---

## 3. Drift to keep an eye on

These are *not* failures; they are watch items. Founder decides if and
when to act.

| Watch item | Why it matters | How to detect |
|---|---|---|
| Stale `chintu-*-report.md` artifacts in repo root | They are gitignored but accumulate locally; harmless | `git status --short` |
| Tracked HTML dashboards drifting from the source command map | A founder might forget to regen after editing the map | Run master launcher |
| New scripts not added to the command map | Silently break the launcher | `node scripts/chintu-command-map.test.js` |
| New docs without cross-links | Reduce discoverability | `node scripts/chintu-doc-link-integrity.test.js` |
| New generators not added to the generated-files map | Silently break this hygiene model | `node scripts/chintu-generated-files-map.test.js` |
| Memory vault files not listed in vault README | Test will catch it | `node scripts/chintu-memory-vault.test.js` |

---

## 4. Out-of-scope (do not "fix" without founder approval)

- Renaming or relocating `CHINTU_*` docs (breaks every cross-link).
- Squashing or rewriting committed history.
- Removing legacy bridge / iMac packages (founder may still want them).
- Touching `.gitignore` rules around BALA assets.
- Editing protected BALA app files (`app.js`, `index.html`,
  `styles.css`, `sw.js`, `coach.js`, `manifest.webmanifest`,
  `privacy.html`, `functions/api/coach.js`).

---

## 5. Suggested slow-day cleanups (each is one safe commit)

These are *optional*. Each one is small, reviewable, and reversible.

1. Add `CHINTU_GENERATED_FILES_MAP.md` to the doc-link integrity test's
   coverage by linking to it from `CHINTU_START_HERE.md` and
   `CHINTU_ARTIFACT_POLICY.md`.
2. When the next test is added, update `CHINTU_FOUNDER_COMMAND_MAP.md`
   *in the same commit* — the command-map test will catch it anyway.
3. Periodically grep the repo for `TODO:` and `FIXME:` and convert each
   into a memory vault entry or a parked-system note.
4. Trim the trailing `-report.md` artifacts before a founder demo so
   `git status` reads as truly clean.
