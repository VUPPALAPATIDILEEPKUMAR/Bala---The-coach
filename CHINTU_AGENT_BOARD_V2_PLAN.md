# Chintu Agent Board V2 Plan

**Stage:** 20
**Status:** planning — next implementation target
**Mode:** local-first multi-agent OS scaffold

---

## What Changes in V2

Agent Board V1 (Stage 17) established the concept: specialist packets, dry-run only,
founder-reviewed before execution.

V2 evolves the board into a real operational surface:

| V1 | V2 |
|---|---|
| 5 agents (static packets) | 8+ agents with role packets |
| Packet = planning doc only | Packet = planning + last-run summary |
| No run history | CHINTU_AGENT_RUNS/ tracks runs per agent |
| No prompt generation | Prompt Engine feeds packets |
| Manual status | Latest-run pointer surfaced in console |

---

---

## V2 Packet Structure (per agent)

Each packet folder in `CHINTU_AGENT_PACKETS/` will include:
- `{agent}.packet.md` — mission, files, allowed/forbidden actions
- `{agent}.prompt.xml` — generated by prompt engine (Stage 20+)
- `{agent}.last-run.md` — summary of last dry-run (written by runner)

---

## V2 Board Output

The agent board script will surface:
- Per-agent status: packet present / last-run present / last-run date
- Total agents configured vs. total run
- Recommended next agent to run based on planner queue

---

## V2 Console Integration

The operator console will show:
- Agent board summary table
- Latest run per agent (from CHINTU_AGENT_RUNS/)
- "Next recommended agent" based on action queue priority

---

## Implementation Order

1. **Stage 20 (this stage):** Prompt Engineer Agent packet + prompt engine script
2. **Stage 21:** Latest-run pointer in operator console
3. **Stage 22:** Product Strategist + Release Manager packets
4. **Stage 23:** Multi-agent coordination prompt generation

---

## Safety Rules (V2, same as V1)

- Packets are bounded. Agents cannot exceed their allowed files.
- No external messaging, no webhook, no connector activation inside packets.
- No secrets in packets or prompts.
- No health data exposed in agent runs.
- Dry-run remains default for all agents.
