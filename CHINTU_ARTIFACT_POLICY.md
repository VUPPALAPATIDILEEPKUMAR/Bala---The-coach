# Chintu — Artifact Policy

Which files are written by hand and which are regenerated by scripts.
This exists so a dirty working tree is easy to read.

---

## Generated, but tracked (review the diff before committing)

These are regenerated by scripts, but kept in git so the dashboard and
briefing surfaces are visible without running anything. Small diffs
(timestamps, commit hashes) are expected after every run. Larger
structural diffs deserve a quick look before committing.

| File | Generator |
|---|---|
| `CHINTU_AGENT_DASHBOARD.html` | `scripts\chintu-agent-dashboard.ps1` |
| `CHINTU_CONTROL_ROOM_INDEX.html` | `scripts\chintu-control-room-index.ps1` |
| `CHINTU_OS_HEALTH_CHECK.md` | `scripts\chintu-os-health-check.ps1` |
| `CHINTU_ALIVE_BRIEFING.md` | `scripts\chintu-alive-briefing.ps1` |

Rule of thumb: if the only diff is a `Generated:` timestamp or the last
commit hash, you can either commit it or leave it; both are fine. If
sections are added or removed, treat that as a real change.

---

## Hand-authored source

Everything else. The big categories:

- `CHINTU_*.md` operator docs (start-here, when-stuck, command map,
  handoffs, sprint plans, etc.)
- `BALA_*.md` BALA-side planning docs (no app code edits)
- `CHINTU_MEMORY_VAULT/**.md` durable founder context
- `CHINTU_IMAC_PACKAGES/**` packages prepared for the iMac side
- `scripts/chintu-*.ps1` and `scripts/chintu-*.test.js` operator code
- Protected BALA app files (`app.js`, `index.html`, `styles.css`,
  `sw.js`, `coach.js`, `manifest.webmanifest`, `privacy.html`,
  `functions/api/coach.js`) — only the founder edits these.

---

## When in doubt

If you cannot tell whether a dirty file is generated:

```powershell
git log -1 -- <path>
```

If the most recent change was a `chore:` commit from a Chintu sweep and
the only diff is a timestamp, it is generated. Otherwise treat it as
source.
